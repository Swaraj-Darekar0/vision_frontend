# Session History Synchronization Documentation

This document outlines the database schema requirements and the API communication pattern necessary to ensure that a user's speaking history and detailed analysis results are persisted in the cloud and can be fully restored to any device.

## 1. Database Schema (`session_scores` table)

To support full synchronization, the Supabase `session_scores` table must include both the calculated metrics and the metadata required to reconstruct the UI.

### Table Definition (SQL)
```sql
ALTER TABLE public.session_scores 
ADD COLUMN topic_title text DEFAULT 'Untitled Session',
ADD COLUMN duration_label text DEFAULT '--',
ADD COLUMN is_first_session boolean DEFAULT false,
ADD COLUMN raw_result jsonb; -- Stores the complete EvaluationResult JSON
```

### Attribute Details

| Attribute | Type | Why it is required |
| :--- | :--- | :--- |
| `session_id` | `text` | Primary key. Matches the UUID generated by the analysis server. |
| `user_id` | `uuid` | Foreign key to `user_profiles`. Ensures users only see their own data. |
| `session_date` | `timestamptz` | Used to sort the history list (Newest first) and render the "Mar 10" labels. |
| `overall` | `real` | Used for the color-coded "Streak Grid" on the Dashboard. |
| `confidence` | `real` | Displayed in the Dashboard performance cards. |
| **`topic_title`** | `text` | **New.** The name of the topic presented (e.g., "Leadership Under Pressure"). |
| **`duration_label`** | `text` | **New.** The formatted duration string (e.g., "05:22") shown in history rows. |
| **`is_first_session`**| `boolean`| **New.** Essential for the backend's "Progress Comparison" logic. |
| **`raw_result`** | `jsonb` | **New.** Stores the full `EvaluationResult` JSON. Without this, the Detail View cannot be reconstructed after a sync. |

## 2. Frontend to Backend Request (`POST /analyze/full`)

When the frontend completes a recording, it sends a `multipart/form-data` request to the analysis server. This payload now includes the metadata necessary for the backend to populate the columns listed above.

### Sample Request Profile

**Endpoint:** `https://vision-backend-evzk.onrender.com/analyze/full`  
**Method:** `POST`  
**Content-Type:** `multipart/form-data`  
**Auth:** `Bearer <supabase_access_token>`

### Multipart Payload Data

| Key | Value Type | Description |
| :--- | :--- | :--- |
| `video` | `File (mp4)` | The 480p compressed video file. |
| `user_id` | `string (uuid)` | The unique ID of the authenticated user. |
| `topic_title` | `string` | The title of the presentation topic. |
| `duration_label` | `string` | Formatted duration (e.g., "01:05"). |
| `is_first_session` | `string ("true"|"false")` | Indicates if the user has no previous history. |

## 3. Sync Flow Logic

1. **Backend Role:** Upon finishing analysis, the backend should save the final `EvaluationResult` JSON into the `raw_result` column along with the provided metadata.
2. **Frontend Role:** When the user clicks the "Sync" button in the History screen, the app calls `syncSessionsFromCloud`. 
3. **Data Restoration:** The app pulls the `raw_result` from the database and populates the local cache (`AsyncStorage`), making the history available offline immediately.
